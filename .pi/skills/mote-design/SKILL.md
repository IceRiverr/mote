# Mote 设计伙伴

> 设计新功能、API、数据结构或模块边界。对话式轻量设计，按需输出文档。

---

## 何时触发

- 用户说 **"design xxx"** → 完整功能设计
- 编码中遇到高 stakes 决策 → 轻量快速决策（不强制出文档）
- 用户说 **"记录一下刚才的设计"** → 事后补写简洁文档

---

## 核心原则

**设计是编码的一部分，不是编码的前置条件。**

大多数决策在对话中快速解决。只有当涉及多文件协调、新模块创建、或重大架构变更时，才走完整设计流程。

---

## 两种模式

### 模式 A：完整功能设计

触发：用户明确说 "design viewport 交互" / "设计 prefab 编辑器"

流程：
```
1. 快速约束收集（2-3 个问题，1 分钟）
2. 列出关键决策点，给出 2-3 个选项
3. 用户确认后，确定文件结构
4. 按需输出 design-xxx.md（用户要求时）
```

### 模式 B：编码中快速决策

触发：用户问 "这个 id 怎么办？" / "entity 要不要 name？"

流程：
```
1. 一句话澄清问题本质
2. 列出 2-3 个选项 + 利弊
3. 用户选一个
4. 简短确认，不强制出文档
```

---

## 快速约束收集（5 个问题）

任何设计开始前，快速确认：

1. **生命周期**：短期（每帧）还是长期（会话级）？
2. **数据归属**：这个数据属于引擎还是编辑器？需要序列化吗？
3. **变更模式**：读多写少？谁写谁读？
4. **已有模式**：找到 2 个最相似的功能。它们怎么解决的？
5. **范围**：影响多少文件？是新模块还是现有文件扩展？

如果任何答案是"不确定"，停下来。陈述不确定性。问用户。

---

## 编辑器数据层硬约束（关键）

编辑器开发占项目 70%+ 工作量，数据层是最易出错的地方。

### 运行时 vs 持久化

| 数据 | 处理方式 |
|------|----------|
| 运行时-only | 不序列化（如 entity 运行时 `id`、选中状态） |
| 持久化 | 序列化到文件（如 `name`、transform、overrides） |
| 运行时推导 | 加载时重新计算（如 `prefab_1`、`prefab_2` 编号） |

### 序列化铁律

- **所有保存入口复用统一序列化函数**（如 `io.ts` 的 `sceneToJson`）
- **禁止多处各自实现序列化**（这是 bug 的首要来源）
- **默认值不在 JSON 中显式出现**（`parent: null`、`visible: true` 应被过滤）
- **运行时数据绝不写入文件**

### 命名生成

同类对象唯一名称，用**最大编号 + 1**，不是**实体数量 + 1**：

```typescript
// ✅ prefab_1, prefab_9 → 最大 9 → prefab_10
// ❌ prefab_1, prefab_9 → 数量 2 → prefab_3（错误）
```

## 歧义处理

遇到高 stakes 决策时（两套架构、与现有模式冲突、破坏性操作）：

**STOP。** 用一句话命名歧义。列出 2-3 个选项及利弊。问用户。不要猜测。

---

## 设计检查清单

最终确定前快速确认：

- [ ] 需要序列化的数据是纯对象 + `version` 字段
- [ ] 运行时-only 数据明确标记，不写入文件
- [ ] 所有保存入口使用统一的序列化函数
- [ ] 编辑器状态与引擎状态分离
- [ ] 破坏性变更时，有明确的废弃/迁移策略
- [ ] 文件结构清晰（新文件放哪？现有文件改哪？）

## 输出格式

### 轻量模式（默认）

对话中直接解决，不输出文件。格式：

```
问题：xxx
选项：
  A. ...（利/弊）
  B. ...（利/弊）
  C. ...（利/弊）
推荐：B，因为 ...
```

### 完整模式（用户要求时）

输出到 `designs/`：

```
design-{关键词}-{YYYYMMDD}.md
```

内容：
```
## 问题
1-2 句话

## 约束
（5 个问题的答案）

## 关键决策
- 决策 X：选了 A（替代 B/C，理由 ...）

## 数据结构
（TypeScript 接口）

## 文件结构
- 新文件：...
- 修改文件：...

## 开放问题
（还需要用户输入什么）
```

## 设计文档维护（重要）

design 文档是**编码时的决策快照**，不是实时维护的架构说明书。

### 不要实时更新

- 设计在对话中持续演进，文档永远追不上
- 维护成本远高于价值，会打断编码节奏
- **代码是唯一真相来源**

### 什么时候更新

| 时机 | 做法 |
|---|---|
| **编码中** | 不碰 design。决策在对话中解决，代码即文档。 |
| **功能完成** | 如 design 严重偏离，在文档底部追加「修订记录」 |
| **长期维护** | 如 design 完全过时，写新文档，保留旧的做历史参考 |

### 修订记录格式

```markdown
## 修订记录

2026-04-19: 编码中发现「编辑器只是预览器」，去掉 entity 持久化 id
- 原设计：entity.id 序列化到文件
- 变更：id 改为运行时-only
- 影响文件：Scene.ts, io.ts, SceneFS.ts
```

这比重写整个 design 轻量 10 倍。

---

## 设计推翻处理

用户可能在编码中途推翻设计：

1. **列出影响面**：哪些文件已按旧设计实现？
2. **标记废弃代码**：旧文件、旧函数、旧字段
3. **明确清理策略**：兼容还是拒绝？是否需要删除旧文件？
4. **验证一致性**：新格式保存 → 新格式加载 → 行为正确

---

## 常见设计陷阱

| 陷阱 | 避免方式 |
|------|----------|
| 直接 `as T` 类型断言 | 用 `sceneFromJson(json)` 做运行时转换 |
| 多处实现序列化 | 所有保存入口复用 `io.ts` 的函数 |
| 用实体数量做编号 | `getNextEntityName` 基于最大编号 |
| 运行时 id 写入文件 | 明确标记运行时-only 字段，序列化时过滤 |
| 旧架构文件残留 | 推翻设计后检查废弃文件，确认无引用后删除 |
