# Mote 讨论笔记

> 把讨论中的要点和轻量决策落到磁盘，避免事后忘记。永远只追加，不修改。

**Tradeoff:** 不结构化、不归类、不删除。原话第一，便于将来重读时还原当时的语境。

## 1. 触发条件

明确触发：用户说出以下任一关键词
- 加到笔记 / 加到 note / 加到 note
- 记一下 / 记下来 / 先记着
- 这点重要 / 这个要记 / 存档一下
- note this / add to note / save this / remember this

**触发条件不明确时，停下来问用户。不要猜。**

## 2. 行为流程

### 步骤 1：确认 slug

按以下顺序判断 slug：

1. **当前对话上下文已有 slug**（用户之前在同一会话中已确认过）→ 直接复用
2. **当前对话围绕 features/ 下已存在的某个目录**（用户提及该 feature 名）→ 复用该目录名
3. **都没有** → AI 主动建议一个 kebab-case slug，例如：
   ```
   建议 slug: particle-system，OK 吗？
   ```
   等用户确认或修改。**不要自作主张创建。**

### 步骤 2：确保目录和文件存在

- 若 `features/{slug}/` 不存在 → 创建目录
- 若 `features/{slug}/note.md` 不存在 → 创建空文件

### 步骤 3：决定追加位置

- 读取 note.md 末尾
- 如果文件最后一段标题是**今天**（`## YYYY-MM-DD`，与当前日期相同）→ 追加到该段最后一项下
- 否则 → 文件末尾新开一段，标题为 `## YYYY-MM-DD`

### 步骤 4：写入条目

格式：

```markdown
## YYYY-MM-DD

- 原话："<用户的关键句子>"
- 上下文：<一句话补充背景，可选,只有补充必要时才写>
```

**说明：**
- 原话尽量保留用户的措辞，不要润色、不要"翻译"成正式说法
- 上下文只在用户原话脱离背景看不懂时才加，能让原话独立成立时省略
- 一次追加一条，不合并多条

### 步骤 5：报告

简短告知用户：
```
已追加到 features/{slug}/note.md（YYYY-MM-DD 段）
```

不复述写入的内容。

## 3. 内容范围

mote-note 接受两类内容：

| 类型 | 例子 |
|---|---|
| **讨论要点** | "用 SoA 布局更适合 compute pass" |
| **轻量决策** | "id 用 string 不要 number" / "这个 buffer 先用 256 byte 对齐" |

两类同等对待，统一追加格式。

## 4. 硬约束

- **只追加，不修改**：既有内容一个字符都不动
- **不归类、不重排**：不按主题分组，不调整顺序
- **不删除**：哪怕用户说"刚才那条不算"，也不删，请用户在新条目里写"撤销 X"
- **不写 design、不写 plan**：超出本 skill 范围
- **不格式化既有 note.md**：哪怕看到旧格式不规范，也不动

## 5. 边界

- 如果用户描述的是**完整设计**（不是单点要点）→ 提示用户考虑用 mote-design
- 如果用户描述的是**任务拆分**（不是要点）→ 提示用户考虑用 mote-plan
- 如果用户没指明要记什么，只说"记一下" → 反问"记哪一点？"，不要把整段对话都塞进去

## 6. 特殊情况

**用户说"刚才那点加到 note"，但"刚才"指代不清楚**：
- 列出最近 2-3 个候选要点，让用户选一个
- 不要默默选最后一句

**用户在两个 feature 之间反复横跳**：
- 每次都要确认当前是哪个 slug，不假设延续上一次的
