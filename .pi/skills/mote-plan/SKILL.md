# Mote 任务拆解

> 把模糊需求变成可落地的功能清单。每个功能是一步可见的进展。

**Tradeoff:** 偏向可见效果而非技术完备。对于纯重构任务，自行判断。

## 1. 按可见功能拆

**不按技术阶段拆。按用户能看到的东西拆。**

❌ 坏的：
```
阶段 1：改类型定义
阶段 2：改序列化逻辑
阶段 3：改 UI 组件
```

✅ 好的：
```
功能 1：SceneTree 显示 entity name
功能 2：Inspector 可编辑 name
```

每个功能包含自己的数据 + 逻辑 + UI，围绕一个可见目标。

**每个功能的标准格式：**
```
功能 N：[一句话可见目标]
  - [子任务 1]
  - [子任务 2]
  → 验证：[打开编辑器后做什么，期望看到什么]
```

**验证方式（按优先级）：**
1. 操作编辑器（有 UI 的功能）
2. 文件检查（序列化/格式变更）
3. 编译通过（纯类型/重构）

## 2. 设计中道变更（常态）

设计在编码中推翻是正常的。处理流程：

- 评估哪些功能已按旧设计完成，哪些未开始
- 已完成的 → 新增"回滚/迁移"子任务
- 未开始的 → 直接按新设计拆解
- **在现有 plan 上打补丁，不重新生成整个计划**
- **以代码为准，不实时更新 design 文档**

## 3. 依赖与顺序

标注依赖：
```
功能 A → 功能 B   (B 依赖 A)
功能 C || 功能 D  (C 和 D 可并行)
```

**用户能最早看到效果的功能优先。**

## 4. 执行纪律

- 一次实现一个功能。验证通过再做下一个。
- 验证失败就停下来，不要跳过。
- 编码中发现设计缺陷 → 标记为"需要 design 调整"，当前功能内修复或暂停。
- "顺便我也应该..." → 记下来，当前 plan 完成后再提。

## 5. 输出

保存到 `plans/plan-{关键词}-{YYYYMMDD}.md`：

```
## 目标
1-2 句话

## 功能清单

### 功能 1：[可见目标]
- [子任务]
→ 验证：[具体操作 + 期望结果]

### 功能 2：[可见目标]
...

## 依赖
功能 1 → 功能 2
```
