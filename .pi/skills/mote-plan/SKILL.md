# Mote 任务拆解

> 把模糊需求变成可落地的功能清单。每个功能是一步可见的进展。

**Tradeoff:** 偏向可见效果而非技术完备。对于纯重构任务，自行判断。

**触发条件不明确时，停下来问用户。不要猜。**

## 1. 前置：必须有 design

`mote-plan` 默认在 `features/{slug}/design.md` 已存在的前提下运行：

- 若 `features/{slug}/design.md` 不存在 → 提示用户先用 `mote-design` 输出 design，**不要自己临时编一份**
- 若用户坚持只做轻量重构、不需要 design → 允许跳过，但 plan 顶部需注明 "无 design"
- plan 不读取 notes.md（讨论原话档案与任务拆解无关，避免噪声）

## 2. 按可见功能拆

**不按技术阶段拆。按用户能看到的东西拆。**

❌ 坏的：
```
阶段 1：改类型定义
阶段 2：改序列化逻辑
阶段 3：改 UI 组件
```

✅ 好的：
```
功能 1：SceneTree 显示 entity name
功能 2：Inspector 可编辑 name
```

每个功能包含自己的数据 + 逻辑 + UI，围绕一个可见目标。

**每个功能的标准格式：**
```
功能 N：[一句话可见目标]
  - [子任务 1]
  - [子任务 2]
  → 验证：[打开编辑器后做什么，期望看到什么]
```

**验证方式（按优先级）：**
1. 操作编辑器（有 UI 的功能）
2. 文件检查（序列化/格式变更）
3. 编译通过（纯类型/重构）

---

## 3. 前置条件检查（编码前必做）

每个功能开始前，先检查以下问题：

| 检查项 | 说明 |
|--------|------|
| **store 是否已有数据？** | 新功能依赖的 store 是否已有初始化/加载机制？ |
| **标识符体系是否一致？** | 不同模块引用同一资源时用的键是否统一？ |
| **数据格式是否兼容？** | 磁盘 JSON 格式与代码中的 TS 类型是否一致？ |
| **外部依赖是否就绪？** | 需要的外部文件（图片、字体、wasm）是否存在？ |

**如果发现基础设施缺失，立即把它提升为独立功能。**

示例：
```
功能 3：实体显示真实精灵图片
  ⚠️ 前置检查发现 spriteSheets store 在项目初始化时未被填充
  → 先完成功能 2.5：SpriteSheet 自动加载扫描
```

---

## 4. 设计中道变更（常态）

设计在编码中推翻是正常的。处理流程：

- 评估哪些功能已按旧设计完成，哪些未开始
- 已完成的 → 新增"回滚/迁移"子任务
- 未开始的 → 直接按新设计拆解
- **在现有 plan 上打补丁，不重新生成整个计划**
- **以代码为准，不实时更新 design 文档**

**遇到高 stakes 变更时（如坐标系翻转、标识符体系重构）：**
- STOP，列出 2-3 个选项及利弊
- 用户确认后，在 plan 底部追加「执行中修订」记录
- 如需搁置，在 features/{slug}/plan.md 末尾标记为待决策段落

---

## 5. 依赖与顺序

标注依赖：
```
功能 A → 功能 B   (B 依赖 A)
功能 C || 功能 D  (C 和 D 可并行)
```

**隐藏依赖的处理：**
编码中发现 plan 未标注的依赖时：
1. 在 plan 文件底部追加「执行中修订」记录
2. 把缺失的基础设施提升为独立功能（见第 3 节）
3. 继续当前功能前先完成基础设施功能

**用户能最早看到效果的功能优先。**

---

## 6. 执行纪律

- 一次实现一个功能。验证通过再做下一个。
- 验证失败就停下来，不要跳过。
- 编码中发现设计缺陷 → 标记为"需要 design 调整"，当前功能内修复或暂停。
- **"顺便我也应该..." → 记下来（建议走 mote-note），判断是否需要提升为独立功能。**
- 验证步骤不够具体 → 停下来完善验证步骤再继续。

---

## 7. 验证步骤规范

验证必须具体到可操作：

```
→ 验证：
  1. 打开浏览器访问 http://localhost:5174
  2. [具体操作：点击什么 / 输入什么 / 按什么键]
  3. 期望看到：[具体 UI 变化 / 数据变化]
  4. 不应看到：[错误日志 / 警告 / 空白/占位符]
```

**反面检查必填：**
- Console 无红色错误
- 无 `[xxx] not found` 类警告
- 功能切换后无残留状态（如模式切换后拖拽不卡死）

---

## 8. Slug 与目录

进入拆解前先确认 slug：

1. 用户已在当前会话中确认过 slug（如和 mote-design 同一会话）→ 复用
2. 用户描述的 feature 在 `features/` 下已存在同名目录 → 复用
3. 都没有 → AI 主动建议一个 kebab-case slug，等用户确认

plan 文档落到 `features/{slug}/plan.md`。

---

## 9. 输出

保存到 `features/{slug}/plan.md`：

```
## 目标
1-2 句话

## 功能清单

### 功能 1：[可见目标]
- [子任务]
→ 验证：[具体操作 + 期望结果]

### 功能 2：[可见目标]
...

## 依赖
功能 1 → 功能 2

## 执行中修订
[日期]: [问题描述]
- 原计划：[xx]
- 实际发现：[xx]
- 调整：新增功能 X / 修改验证步骤 / 搁置
```

## 10. 常见陷阱（基于真实编码经验）

| 陷阱 | 示例 | 对策 |
|------|------|------|
| **假设 store 已有数据** | 功能"显示精灵图片"假设 `spriteSheets` 已加载，实际为空 | 编码前检查 store 初始化机制 |
| **标识符不一致** | Prefab 用文件名引用 atlas，store 用 `id` 做键 | 前置检查时对比引用链 |
| **数据格式漂移** | JSON 存 `{shapes: [...]}`，代码期望 `ColliderShape[]` | 编码前对比一个真实样本 |
| **验证只看"正面"** | 只验证"图片显示"，不验证"无报错" | 验证步骤必须包含反面检查 |
| **功能粒度太大** | "渲染精灵"实际包含加载+解析+绘制+错误处理 | 拆分为"加载数据"+"绘制"两个功能 |
| **忽视文件系统时序** | Canvas 在 SpriteSheet 加载完成前就开始渲染 | 验证步骤要求按正确顺序操作 |
