# Mote 任务拆解与执行计划

> 把设计决策变成可执行步骤。设计可以是演进的——中途推翻假设时，计划跟着调整。

---

## 何时使用本技能

- 设计文档就绪，需要开始编码
- 大功能需要拆成增量交付物
- 需要建立里程碑与验收标准
- 多文件/模块需要协调变更
- 用户问"怎么开始？""从哪下手？"

---

## 歧义处理协议

遇到高风险的歧义时（两套可能的架构、与现有模式冲突、破坏性操作、关键上下文缺失）：

**STOP。** 用一句话命名歧义。列出 2-3 个选项及利弊。问用户。不要在架构或数据模型决策上猜测。

不适用于：常规编码、小功能、显而易见的变更。

---

## 计划流程

### 两种起点

**起点 A：设计已明确**
- 用户提供了清晰的设计方向
- 直接拆解任务，按阶段执行

**起点 B：设计需探索**
- 用户说"plan xxx"但方向模糊
- 先用 2-3 个快速问题澄清约束
- 拆解任务时标注"未决设计点"

### 阶段 1：拆解

把设计拆成**原子任务**。一个原子任务：

- 最多触及 2-3 个文件
- 能用一句话描述
- 有明确的 yes/no 完成标准
- 大约 15-60 分钟专注实现

**如果任务太大，继续拆分。** 如果太小（5 分钟），和相关任务合并。

### 阶段 2：依赖排序

按依赖排序：

1. **地基先行**：数据结构、类型、接口、常量
2. **核心逻辑**：系统、算法、业务规则
3. **集成接线**：事件绑定、UI 连接、命令串联
4. **清理废弃**：删除旧文件、移除废弃引用、统一序列化入口
5. **打磨**：样式、动画、边界情况、错误提示

依赖标注：
```
任务 A → 任务 B      (B 依赖 A)
任务 C || 任务 D     (C 和 D 独立，可并行)
```

### 阶段 3：验证定义

每个任务定义**怎么知道它做完了**。验证方式取决于任务类型：

| 任务类型 | 验证方式 |
|---|---|
| 数据/类型变更 | TypeScript 编译零错误 |
| 序列化逻辑 | 保存文件 → 人工检查 JSON 内容 |
| UI 组件 | 视觉检查 + 交互检查 + undo/redo 检查 |
| 命令/Store | 执行操作 → 观察状态变化 |
| 引擎 System | 单元测试或手动测试 |
| 性能优化 | 测量前后对比 |

**编辑器开发最常见的验证：**
- "保存场景，检查文件无运行时 id、无默认值冗余"
- "刷新页面加载场景，确认所有 entity 正确显示"
- "SceneTree 点击选中，Inspector 实时更新"
- "编译零错误" |

### 设计中道变更处理

当用户中途推翻设计假设（如"id 不要持久化了""编辑器只是预览器"）：

1. **评估影响**：哪些任务已按旧设计完成？哪些任务还未开始？
2. **调整计划**：
   - 已完成的任务 → 新增**回滚/迁移任务**
   - 未开始的任务 → 直接按新设计拆解
3. **新增清理任务**：删除旧代码、统一序列化、验证无引用
4. **不强制重新生成整个计划**——只做增量调整

**示例：**
```
原计划：
  1. 设计持久化 entity id → 已完成 ❌
  2. 实现 id 生成逻辑 → 已完成 ❌
  3. 序列化 id 到文件 → 未开始

用户推翻："id 不要持久化"

调整后：
  1. [回滚] 移除 entity 的持久化 id → 验证：scene 文件无 id 字段
  2. [回滚] 修改 createSceneEntity 不再接受 options.id → 验证：编译零错误
  3. [新增] 加载时重新生成运行时 id → 验证：旧文件仍可加载
  4. [新增] 清理 io.ts 的 id 序列化逻辑 → 验证：保存文件无 id
```

## 任务类别

### 🏗️ 地基（数据与类型）

- 定义接口、类型、枚举
- 创建 Component 类（无参构造函数！）
- 添加常量、配置对象
- 更新注册表（ComponentRegistry、System registry 等）

**验证**：TypeScript 编译无错误。新类型可导入。

### ⚙️ 核心（逻辑与系统）

- 实现 System 更新循环
- 实现算法、工具函数
- 添加引擎插件初始化

**验证**：单元测试或手动测试通过。已知输入产生预期输出。

### 🔌 集成（接线）

- 连接新系统到游戏循环
- 钩子：编辑器 UI ↔ 引擎状态
- 添加事件监听 / 命令绑定
- 更新模块导出 / barrel 文件

**验证**：端到端：执行用户操作 → 观察正确结果。

### 🧹 清理（废弃与统一）

- 删除旧架构遗留文件
- 移除废弃引用和 import
- 统一序列化入口（所有保存走 `io.ts`，禁止重复实现）
- 验证无编译错误、无运行时残留

**验证**：
- `grep` 确认旧类型/函数无引用
- 编译零错误
- 保存文件人工检查格式一致性

### 🎨 UI/UX（编辑器）

- 添加 Preact 组件
- 连接 Signals 到引擎数据
- 实现命令（execute/undo/redo）
- 添加键盘快捷键

**验证**：视觉检查 + 交互检查 + undo/redo 检查。

### 🧪 测试与打磨

- 边界情况测试
- 性能验证（帧时间、内存）
- 序列化往返测试
- WebGL2 fallback 检查（如涉及图形）

## 输出格式

```
## 概览
总任务数：N
预计阶段：N
关键路径：任务 X → 任务 Y → 任务 Z

## 阶段 1：地基
1. [任务名] → 验证：[具体标准]
2. [任务名] → 验证：[具体标准]

## 阶段 2：核心逻辑
3. [任务名] → 验证：[具体标准]
...

## 阶段 3：集成
...

## 阶段 4：清理废弃
...

## 阶段 5：打磨
...

## 风险区域
- [可能出错的地方及缓解措施]

## 未决问题
- [可能阻塞执行的事项]
```

保存到项目根目录 `plans/`：

### 文件命名

```
plan-{关键词}-{YYYYMMDD}.md
```

- `关键词`：2-4 个描述需求的关键词，小写，短横线分隔
- `YYYYMMDD`：当前日期

示例：
- `plan-health-damage-system-20260418.md`
- `plan-particle-renderer-webgpu-20260418.md`

## 执行指南

### 编码中

- **一次一个阶段。** 阶段 1 验证通过前不开始阶段 2。
- **如果任务暴露设计缺陷，暂停。** 不要绕过它打补丁。标记它，提出设计调整，等确认。
- **每阶段结束后提交。**（如果用版本控制。）
- **验证失败就调试，不要"以后再说"。**

### 处理阻塞

1. 一句话说明阻塞
2. 列出 2-3 种解决方案及利弊
3. 问用户选哪个
4. 解决前不继续

### 范围蔓延防御

如果编码中发现"顺便我也应该..."：

- 记下来作为后续任务
- **不要在当前任务里实现**
- 当前计划完成后再提给用户

### 设计演进处理

设计不是一锤子买卖。实际模式：

```
用户说"plan xxx" → 拆解任务 → 编码中 → 用户推翻假设 → 调整计划 → 继续编码
```

**关键原则：**
- 不要等"设计最终化"才开始拆解
- 把"未决设计点"作为计划中的显式风险
- 用户推翻时，快速评估影响、调整计划、继续推进

**处理流程：**

1. **列出影响面**：哪些文件已按旧设计实现？
2. **标记废弃代码**：旧文件、旧函数、旧字段
3. **新增清理任务**：删除/迁移旧代码
4. **验证一致性**：新格式保存 → 新格式加载 → 旧文件仍可读（兼容）或拒绝加载（破坏性）

## 示例

**设计：** 添加 Health 组件和伤害系统，编辑器支持。

```
## 概览
总任务数：7
预计阶段：4
关键路径：定义组件 → 注册组件 → HealthSystem → Inspector 面板

## 阶段 1：地基
1. 创建 `Health.ts` 组件，`current`、`max` 字段 → 验证：编译通过，构造函数无参
2. `ComponentRegistry` 注册 Health → 验证：`world.add(entity, new Health())` 可用

## 阶段 2：核心逻辑
3. 创建 `HealthSystem.ts`，Damage 事件触发时递减 `current` → 验证：Health=10，Damage=3，结果 7
4. 创建 `Damage` 事件类型 → 验证：可 emit 和 consume

## 阶段 3：集成
5. HealthSystem 加入引擎系统列表 → 验证：每帧运行
6. 编辑器 Inspector 添加 Health 面板 → 验证：选中 entity 显示 current/max，可编辑

## 阶段 4：清理（如旧系统存在）
7. 删除旧的生命值字段 → 验证：编译零错误，旧引用清零

## 阶段 5：打磨
8. Health 编辑支持 undo/redo → 验证：修改 → undo → redo，数值正确

## 风险区域
- Damage 事件分发：如果事件系统不存在，需要先创建它（新增地基任务）
- Inspector 可能需要数字输入组件；检查 shared-ui 中是否已有
```
